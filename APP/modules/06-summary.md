## 1. Purpose & Scope

The Summary Tab provides AI-generated summaries for a recording. Users can view multiple summary versions (default template, One-pager, custom template), switch between them, edit content, regenerate with different templates/languages, export to PDF/Word, and share via link or third-party integrations (Notion, Slack). This tab also hosts the Task extraction sub-module.

**In-scope**: Summary list management, template selection, summary generation trigger, HTML rendering in WebView, summary editing, export (PDF/Word/Notion/Slack), Task CRUD with Google Calendar sync, summary language preference.

**Out-of-scope**: Transcript rendering (see `05-transcript.md`), AI Chat (see `07-ai-chat/`), template community browsing UX (separate module).

**Cross-repo boundary**: Summary generation is a 3-repo flow:

```
APP (TranscriptionSummaryController)
  --> BACKEND audio-api POST /api/v1/records/{id}/summary
    --> AI FastAPI POST /records/{audio_id}/summary
      --> AI Celery: parallel LLM calls (default + OnePager + custom)
      --> AI stores reports in PostgreSQL
    <-- AI returns {summaries[]}
  <-- BACKEND returns AES-encrypted response
APP renders HTML in WebView
```

## 2. State Model

### 2.1 Summary Generation State Machine

```
stateDiagram-v2
    [*] --> not_generated : recording processed, no summary yet
    not_generated --> generating : user selects template + taps Generate
    generating --> complete : AI returns summary content (ReportStatus.success)
    generating --> failed : AI pipeline error (ReportStatus.failed)
    failed --> generating : user taps Retry
    complete --> generating : user taps Regenerate (new template/language)
    complete --> editing : user enters edit mode
    editing --> complete : user saves edits (PUT to server)
    editing --> complete : user cancels edits
```

### 2.2 Per-Summary Item Status (ReportStatus enum)

| Status | Code | Meaning |
|--------|------|---------|
| `processing` | 0 | Summary is being generated by AI |
| `success` | 1 | Summary available and renderable |
| `failed` | 2 | Generation failed |

### 2.3 Summary List Management

```
- summaryList: RxList<SummaryItem> -- ordered list of all summary versions
- selectedSummaryIndex: RxInt -- currently displayed tab
- Special indices:
  - [0..N-1]: SummaryItem tabs (default, OnePager, custom...)
  - [N]: Recording Notes tab (if recording has notes from device)
  - [N+1]: Inner Transcript view (quick switch without leaving Summary Tab)
```

### 2.4 State Ownership

| State | Owner | Storage |
|-------|-------|---------|
| Summary list | `TranscriptionSummaryController.summaryList` | `TranscriptionCacheService` (Hive) + server |
| Generation status | `TranscriptionSummaryController.isLoadingSummaries` | In-memory reactive |
| Selected language | `SummaryLanguageController` (global singleton) | Server + local preference |
| Task list | `TranscriptionTaskController` | Server + local cache |

## 4. Data Contract

### 4.1 Core Models (APP side)

```dart
// SummaryItem
{
  id: String,                 // summary ID (report ID)
  status: ReportStatus,       // processing(0) | success(1) | failed(2)
  format: SummaryFormat,      // "markdown" | "html" | "one_pager"
  template_name: String,      // display name of template used
  content: String,            // HTML content for WebView rendering
  actionItemsMapping: Map,    // action item to transcript line mapping
  version: int,
  report_id: String,
  created_at: String,
  updated_at: String
}

// SummaryLanguageOption
{
  code: String,          // e.g. "en", "zh-CN"
  appName: String,       // display name in app locale
  nativeName: String,    // name in native language
  notes: String?
}

// SummaryLanguagePayload (from API)
{
  recent: List<SummaryLanguageOption>,  // recently used languages
  all: List<SummaryLanguageOption>      // all supported languages
}
```

### 4.2 API Endpoints

| Endpoint | Method | Direction | Purpose |
|----------|--------|-----------|---------|
| `/api/v1/records/{id}/summary` | POST | APP->BE->AI | Trigger summary generation `{template_id, summary_language?}` |
| `/api/v1/records/{id}/report/regenerate` | POST | APP->BE->AI | Trigger full regeneration `{template_id, summary_language}` |
| `/api/v1/exports/{reportId}` | POST | APP->BE | Start export task `{is_transcription, target_format}` |
| `/api/v1/exports/{taskId}` | GET | APP->BE | Poll export status |
| `/api/v1/users/supported-languages` | POST | APP->BE | Fetch supported summary languages |
| `/api/v1/users/used-language` | POST | APP->BE | Save language preference |
| `/api/v1/shares` | POST | APP->BE | Create share link for summary+transcript |

### 4.3 AI Backend Summary Generation

| Step | Input | Output | Transformation |
|------|-------|--------|---------------|
| Language normalization | `summary_language` code or null | Human-readable name | `from_lang_code_to_name()`: "en"->"English", null->auto-detect from `conversation.language_code` |
| Template resolution | `template_ids[]` | Template content from `template_locales` | Load locale-specific template instructions |
| Parallel generation | transcript lines + template + language | 2-3 summaries | `asyncio.gather(default_md, onepager_json, custom_md)` |
| Post-processing | LLM markdown output | HTML + source links | `markdown_to_html()`, embedding similarity for source tracing, `memoket://` deep links |

## 5. Key Scenarios

### Scenario 1: Generate Summary with Custom Template

```
1. User taps "+" (Add Summary) on Summary Tab
2. APP checks network -> OK
3. APP loads template list from Template Community API (with topic-based recommendation)
4. User browses/searches templates, selects one
5. APP sends POST /api/v1/records/{id}/summary {template_id: custom_id, summary_language: "en"}
6. BACKEND forwards to AI with template_ids: [default, onepager, custom_id]
7. AI runs 3 parallel LLM generations
8. AI stores 3 Report rows, returns summaries
9. APP receives response, adds to summaryList, renders in WebView tabs
```

### Scenario 2: Regenerate Summary (Ticket-001493 relevant)

```
1. User taps Regenerate menu item
2. RegenerateWarningBottomSheet displayed -> user confirms
3. GenerateTranscriptionBottomSheet shown -> user selects template + language
4. TranscriptionController._rebuildRecord() called
5. POST /api/v1/records/{id}/report/regenerate sent
6. BACKEND deletes old data, triggers AI rebuild (full Celery chain)
7. APP enters isGenerating = true state, shows processing animation
8. AI completes -> notifies via Redis Stream -> BACKEND PushWorker -> SSE/FCM -> APP
9. **Critical path**: RecordingRepository.rxItems listener detects status=audioProcessed
   -> calls _fetchAllDataForProcessed() -> refreshes summaryList
10. UI shows new summary content
```

**Known bug (Ticket-001493)**: The Repo listener previously only set `isGenerating=false` without calling `_fetchAllDataForProcessed()`, causing the UI to exit loading but display stale/empty content. Fix: add `_fetchAllDataForProcessed()` call + restore polling compensation.

### Scenario 3: Export Summary to Notion

```
1. User taps Share -> selects Notion
2. APP checks Notion authorization status (IntegrationsController.isNotionAuthorized)
3. If not authorized -> redirect to Notion OAuth flow (see integrations module)
4. If authorized -> POST export with Notion target
5. BACKEND syncs content to user's Notion workspace
6. APP shows success confirmation
```

## 7. Error Catalogue

| Error Code | Trigger | User-Facing Message | Recovery |
|------------|---------|---------------------|----------|
| E-SM-001 | Summary generation failure (LLM timeout or error) | "Summary generation failed. Please try again." | Retry button; language auto-rollback per FR-SM-022 |
| E-SM-002 | **Notification chain broken** (Ticket-001493 root cause) | UI stuck in generating state | **Fix applied**: Repo listener now calls `_fetchAllDataForProcessed()`; polling compensation restored as fallback (10s interval) |
| E-SM-003 | Summary edit save failure (network) | "Failed to save. Changes kept locally." | Local temp file created; retry on network recovery |
| E-SM-004 | Template list load failure | "Unable to load templates. Check connection." | Retry button on template picker |
| E-SM-005 | Export task failure | "Export failed. Please try again." | Re-submit export task |
| E-SM-006 | Third-party export failure (Notion/Slack auth expired) | "Authorization expired. Please reconnect." | Redirect to integrations settings |

### Ticket-001493 Post-Mortem

**Root cause**: After regeneration, the AI backend completed successfully, but the APP notification chain had multiple potential breakpoints (SSE disconnect, PushWorker channel full, FCM delay). The `StatusPollingService.runPollOnce()` had been emptied (no-op), removing the fallback polling mechanism. Additionally, the `RecordingRepository` listener detected the status change but only toggled `isGenerating` without refreshing data.

**Fix**: (A) Add `_fetchAllDataForProcessed()` in Repo listener when status = audioProcessed. (B) Restore `runPollOnce()` implementation with 10s polling interval during `isGenerating=true`. (C) Widen `onExternalStatusCompleted` condition to accept both `audioProcessing` and `audioUploadFontDone`.
